# 동규 프로젝트 분석

## 개요

이 프로젝트는 **KCG 역량 진단**이라는 이름의 정적 웹 기반 설문/진단 페이지다.
백엔드 없이 **HTML / CSS / JavaScript만으로 구성된 프론트엔드 프로토타입** 형태이며,
조직 구성원의 자기 진단 결과를 보여주는 목적의 간단한 웹앱으로 보인다.

## 폴더 구조

분석 대상 폴더: `/Users/kwakdonghyeun/Desktop/동규`

구성 파일:

- `index.html` — 사용자용 설문 페이지
- `admin.html` — 관리자용 결과 조회 페이지
- `script.js` — 설문 문항, 응답 저장, 결과 계산 로직
- `style.css` — 전체 UI 스타일
- `firebase-config.js` — Firebase 앱 / Firestore 초기화 설정
- `firebase-config.sample.js` — Firebase 설정 교체용 템플릿 파일
- `admin-auth-config.js` — 관리자 로그인 아이디/비밀번호 설정 파일
- `firebase-데이터-설계.md` — Firestore 데이터 구조 설계 문서
- `firebase-구현-계획.md` — Firebase 연동 구현 계획 문서
- `firebase-설정-가이드.md` — Firebase 계정/프로젝트 교체 시 인수인계 가이드
- `firestore.rules` — Firestore 보안 규칙 파일
- `firestore-rules-가이드.md` — Firestore 규칙 적용/운영 가이드

## 프로젝트 성격

이 프로젝트는 다음 성격을 가진다.

- **정적 웹 프로젝트**
- **역량 진단/심리 설문형 서비스**
- **사내 진단 도구 또는 MVP/시안 수준의 프로토타입**
- **배포는 가능하지만 데이터 저장은 미완성인 상태**

## 사용자 흐름

### 1. 기본 정보 입력
`index.html`에서 사용자는 다음 정보를 입력한다.

- 이메일 (필수)
- 이름
- 회사명
- 직급
- 부서

이후 `진단 시작` 버튼을 눌러 설문에 진입한다.

### 2. 설문 응답
총 **24개 문항**에 대해 1~5점 척도로 응답한다.

점수 기준:

- 1점: 전혀 그렇지 않다
- 2점: 그렇지 않은 편이다
- 3점: 보통이다
- 4점: 그런 편이다
- 5점: 매우 그렇다

### 3. 결과 계산
응답 완료 후 `결과 보기` 버튼을 누르면,
문항을 3개씩 묶어 총 8개 역량 카테고리의 평균 점수를 계산한다.

### 4. 제출
현재 구현은 Firestore에 실제 응답 데이터를 저장하는 구조다.

## 측정 카테고리

프로젝트는 아래 8개 역량을 측정한다.

1. 감정식별력
2. 패턴인식
3. 결과예측
4. 감정조절·활용
5. 내적동기
6. 낙관성
7. 공감
8. 목적지향

각 카테고리는 3개 문항으로 구성된다.

## 코드 구조 분석

### `index.html`
역할:
- 메인 설문 페이지 UI 제공
- 기본 정보 입력 폼 제공
- 진행률 표시 UI 제공
- `script.js` 연결

특징:
- 모바일/데스크탑 겸용 설문 페이지 구조
- 좌측 설문 / 우측 진행률 카드 레이아웃을 사용하되 모바일에서는 1열로 전환
- 사용자 노출 문구는 존댓말 톤으로 정리
- 기본 정보 입력 순서: 이메일 → 이름 → 회사명 → 직급 → 부서
- 설문 본문은 초기엔 비어 있고 JavaScript로 렌더링됨

### `script.js`
핵심 비즈니스 로직이 들어있다.

포함 내용:
- 24개 질문 배열 (`questions`)
- 8개 카테고리 배열 (`categories`)
- 응답 저장용 객체 (`answers`)
- 설문 시작 함수 `startSurvey()`
- 설문 렌더 함수 `render()`
- 응답 저장 함수 `save()`
- 진행률 갱신 함수 `updateProgress()`
- 결과 계산 함수 `showResult()`
- 제출 함수 `submitResult()`

특징:
- Firestore `surveyResponses` 컬렉션에 직접 저장
- 저장 필드에 `company` 추가
- 사용자 기본 정보 저장 순서: 이메일, 이름, 회사명, 직급, 부서

### `admin.html`
역할:
- 관리자용 데이터 조회 UI
- 전체 응답 수 / 평균 / 최고 점수 / 최근 제출 표시
- 검색 / 점수 필터 기능 제공
- 개별 결과 상세 보기 / 삭제
- 데스크탑 전용 관리자 대시보드 역할
- 로그인 후에만 접근 가능한 관리자 화면 역할

특징:
- 현재는 Firestore의 `surveyResponses`를 조회하는 구조
- 목록 테이블 중심 구조
- 상세보기는 별도 새 창 팝업으로 표시
- 상세 팝업에서 종합 레이더 차트 + 카테고리별 3문항 레이더 차트 표시
- 데스크탑 사용 기준으로 넓은 레이아웃 적용
- 사용자 노출 문구는 관리자 화면 포함 존댓말 톤으로 정리
- 현재는 프론트엔드 임시 로그인 적용
- 로그인 값은 `admin-auth-config.js`로 분리
- 목록/검색/상세에 회사명 필드 반영
- 검색은 이름/이메일 자유검색 + 회사명/직급/부서명 개별 필터 조합으로 조회 가능
- 현재 조회 결과 또는 전체 데이터를 XLSX(엑셀) 파일로 저장 가능
- 추후 Firebase Authentication으로 교체 권장

### `style.css`
역할:
- 입력 폼, 설문 카드, 결과 카드, 진행률 바 등 스타일 정의

특징:
- 전반적으로 깔끔한 카드형 UI
- 설문 페이지 기준 모바일 대응 레이아웃 강화
- 모바일에서는 선택지를 세로형 카드로 표시하도록 개선
- 관리하기 쉬운 단순 CSS 구조

## 현재 상태에서 확인된 핵심 문제

### 1. Firebase 연동은 붙었지만 Firestore 보안 규칙 확인이 필요함
현재 `submitResult()`는 Firestore의 `surveyResponses` 컬렉션에 직접 저장하도록 변경되었다.

다만 현재 동작 여부는 Firebase 설정과 Firestore 보안 규칙에 따라 달라진다.

확인 필요:
- Firestore Database 생성 여부
- 웹앱 config 적용 여부
- Firestore Security Rules 허용 여부

### 2. 관리자 페이지는 Firestore를 읽도록 변경됨
기존 `admin.html`은 localStorage 구조였지만,
현재는 Firestore의 `surveyResponses` 컬렉션을 조회하도록 변경되었다.

즉, 이제 설문 페이지와 관리자 페이지는 같은 데이터 소스를 바라보는 구조다.

### 3. 백엔드가 없는 완전 클라이언트 구조
이 프로젝트는 모든 로직이 브라우저 안에서만 동작한다.

의미:
- 데이터 영속성 낮음
- 브라우저/기기 바뀌면 데이터 공유 안 됨
- 실서비스용으로는 한계 큼

### 4. 관리자 인증은 아직 임시 방식
현재는 `admin-auth-config.js`를 이용한 프론트엔드 임시 로그인이다.

문제점:
- 진짜 보안 시스템은 아님
- 운영 단계에서는 Firebase Authentication과 Firestore Rules 정교화 필요

### 5. 진행 중 상태 복원 기능 없음
응답 도중 새로고침하면:
- 입력 정보 유실
- 응답 데이터 유실

사용성 측면에서 아쉬움이 있다.

## 잘 만든 부분

### 1. 구조가 단순해서 수정이 쉬움
파일 수가 적고 로직이 한 군데에 모여 있어 빠르게 개선 가능하다.

### 2. 카테고리 설계가 명확함
24문항을 8개 역량으로 나누는 구조가 잘 정리되어 있다.

### 3. 결과 화면이 직관적임
평균 점수와 등급 배지가 있어 사용자가 해석하기 쉽다.

### 4. 관리자 화면 컨셉은 이미 잡혀 있음
완성도는 낮아도,
어떤 식으로 관리 페이지를 운영하려는지는 분명하게 보인다.

- 관련 설계 문서는 프로젝트 루트에 md 파일로 함께 관리
- 코드 수정 시 문서도 같이 업데이트

현재 기준 핵심 문서:

- `동규-프로젝트-분석.md`
- `firebase-데이터-설계.md`
- `firebase-구현-계획.md`
- `firebase-설정-가이드.md`

## 현재 반영된 변경 사항

이번 작업까지 아래가 실제 코드에 반영되었다.

1. `firebase-config.js` 추가
   - Firebase 앱 초기화
   - Firestore 연결 설정

2. `index.html` 변경
   - `script.js`를 `type="module"` 방식으로 로드하도록 변경
   - 기본 정보 입력 필드에 `회사명` 추가
   - 입력 순서를 `이메일 → 이름 → 회사명 → 직급 → 부서`로 변경

3. `script.js` 변경
   - Firestore SDK import 추가
   - 제출 시 `surveyResponses` 컬렉션에 `addDoc()` 저장
   - `company` 필드 추가
   - `createdAt`, `updatedAt`, `averageScore`, `answers`, `results` 저장
   - 제출 중 버튼 비활성화 처리 추가

4. `admin.html` 변경
   - Firestore에서 목록 조회하도록 변경
   - 상세 보기 / 개별 삭제 기능을 Firestore 기준으로 변경
   - 최신순 조회(`createdAt desc`) 적용
   - 검색 / 회사명·직급·부서 개별 필터 / 점수 필터 / 통계 카드를 포함한 데스크탑 관리자 UI로 확장
   - 현재 조회 결과를 CSV(엑셀 호환)로 저장하는 버튼 추가
   - 회사명 컬럼 및 상세 정보 반영
   - 상세보기는 우측 패널 대신 새 창 팝업 방식으로 변경

5. Firebase 교체 가능 구조 문서화
   - `firebase-config.sample.js` 추가
   - `firebase-설정-가이드.md` 추가
   - 다른 Firebase 계정으로 이전 시 문서 재작성 없이 config만 교체하는 구조로 정리

6. `index.html` / `style.css` 모바일 최적화
   - 설문 페이지를 모바일/데스크탑 겸용 구조로 개선
   - 진행률 카드와 안내 카드 분리
   - 모바일에서는 선택지를 세로형 UI로 전환

7. Firestore Rules 문서화
   - `firestore.rules` 추가
   - `firestore-rules-가이드.md` 추가
   - 테스트용 / 운영용 적용 방향 정리

## 결론

현재 이 프로젝트는 **아이디어와 화면은 어느 정도 완성되어 있지만, 운영 기준으로는 인증/보안/검증이 더 필요한 상태의 프로토타입**이다.

즉,
- 보여주기용 데모로는 충분히 가능
- 내부 테스트용으로도 일부 사용 가능
- 하지만 실제 운영용으로는 인증/보안/관리 기능이 더 보강되어야 함

향후 가장 먼저 손봐야 할 부분은 **설문 결과 저장 안정화와 관리자 인증 구조 고도화**다.
